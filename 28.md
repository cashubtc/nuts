# NUT-28: Oracle Spending Condition

`optional`

`depends on: NUT-10`

---

This NUT describes an oracle spending condition that allows ecash proofs to be locked to an oracle-attested outcome. Using this spending condition, proofs can only be spent after an oracle publishes a signature attesting to a specific outcome. This enables use cases such as prediction markets, event-based tokens, and oracle-resolved conditional payments.

The oracle signature scheme is designed to be compatible with the [DLC (Discreet Log Contracts) specification](https://github.com/discreetlogcontracts/dlcspecs), allowing Cashu mints to leverage existing DLC oracle infrastructure.

Caution: Applications that rely on oracle resolution must verify that the oracle is trustworthy and check via the mint's [info][06] endpoint that NUT-28 is supported.

Caution: If the mint does not support this type of spending condition, proofs may be treated as regular anyone-can-spend proofs. Applications must ensure that the mint supports a specific kind of spending condition by checking the mint's [info][06] endpoint.

## Motivation

Oracle spending conditions enable powerful use cases where ecash is released based on real-world events. By designing for DLC compatibility, this specification allows:

1. **Reuse of existing oracle infrastructure** - Existing DLC oracles can be used without modification
2. **Standardized signature verification** - Well-tested BIP 340 Schnorr signatures with tagged hashing
3. **Accountability** - Oracle announcements provide cryptographic proof of oracle commitments

## Oracle and Mint Separation

The separation of oracle and mint roles is a fundamental design principle:

### Benefits for Users

- **Privacy preservation** - If the oracle knows "who bought which share", the incentive to manipulate outcomes increases. Anonymous trading via separated mint/oracle allows prediction market participants to trade without revealing positions to the oracle.
- **Reduced rugpull risk** - When the oracle cannot see user positions, the economic incentive for fraudulent attestations is minimized.

### Benefits for Mints

- **No oracle responsibility** - Operating an oracle requires different skills (domain expertise, signing infrastructure, reputation management). This separation allows mints to focus on their core competency.
- **Zero counterparty risk** - When implemented correctly, the mint has no economic exposure to oracle outcomes. The mint simply enforces the oracle's attestation.

## ORACLE Locked Proof

[NUT-10][10] Secret `kind: ORACLE`

If for a `Proof`, `Proof.secret` is a `Secret` of kind `ORACLE`, the oracle announcement and outcome are stored in the secret's tags, and the oracle signature for unlocking the proof is provided in `Proof.witness.oracle_sig`.

Here is a concrete example of a `Secret` of kind `ORACLE`:

```json
[
  "ORACLE",
  {
    "nonce": "da62796403af76c80cd6ce9153ed3746",
    "data": "",
    "tags": [
      ["announcement", "<json_encoded_oracle_announcement>"],
      ["outcome", "YES"]
    ]
  }
]
```

The `announcement` tag contains the JSON-encoded oracle_announcement which includes the oracle public key, event information, nonces, and maturity. The `outcome` tag specifies which outcome this proof is locked to.

## Tags

The following tags define the oracle spending condition:

### Required Tags

- `announcement: <json_str>` - JSON-encoded oracle_announcement. Contains `announcement_signature`, `oracle_public_key`, and `oracle_event` (with nonces, maturity epoch, event descriptor, and event ID). See [Oracle Announcement Format](#oracle-announcement-format).
- `outcome: <str>` - The specific outcome this proof is locked to. MUST be one of the values in the announcement's event_descriptor.

### Optional Tags

Optional P2PK tags from [NUT-11][11] may also be used for additional spending conditions:

- `sigflag: <str_enum[SIG_FLAG]>` - Sets the signature flag
- `pubkeys: <hex_str>` - Additional public keys for multisig verification (_allows multiple entries_)
- `n_sigs: <int>` - Minimum number of public keys providing valid signatures

## Oracle Signature Scheme

The oracle signature scheme follows the [DLC specification](https://github.com/discreetlogcontracts/dlcspecs/blob/master/Oracle.md#signing-algorithm) for maximum compatibility with existing DLC oracle infrastructure.

### Tagged Hash

Signatures use BIP 340 tagged hashing. The `tag_hash` function is defined as:

```
tag_hash(tag) = SHA256(SHA256(tag) || SHA256(tag))
```

Where `||` denotes byte concatenation.

### Signing Algorithm

The algorithm `Sign(sk, message, tag)` is defined as:

```
Sign(sk, message, tag):
  H = tag_hash("DLC/oracle/" || tag)
  m = H(message)
  return BIP340_sign(sk, m)
```

For attestations, the tag is `"attestation/v0"`, so the full tag string is `"DLC/oracle/attestation/v0"`.

### Message Format

The message to be signed is the **UTF-8 encoded outcome string with NFC normalization**. For example, if the outcome is `"YES"`, the message is the UTF-8 bytes of `"YES"`.

### Complete Signing Example

To sign the outcome `"YES"`:

1. Normalize the outcome string to NFC: `"YES"` (already normalized)
2. Compute the tagged hash:
   - `tag = "DLC/oracle/attestation/v0"`
   - `tag_hash = SHA256(SHA256(tag) || SHA256(tag))`
3. Compute the message hash:
   - `m = SHA256(tag_hash || UTF8("YES"))`
4. Sign with BIP 340:
   - `sig = BIP340_sign(sk, m)`

### Verification

To verify an oracle signature:

```
Verify(pk, message, sig, tag):
  H = tag_hash("DLC/oracle/" || tag)
  m = H(message)
  return BIP340_verify(pk, m, sig)
```

Where `pk` is the 32-byte x-only public key from the announcement's `oracle_public_key`.

## Event Descriptor

The event descriptor is contained within the oracle announcement and defines the enumerated outcomes for an event, equivalent to the DLC `enum_event_descriptor`. This is a simple enumeration where each outcome is a UTF-8 string with NFC normalization.

Example for a binary market (within the announcement):

```
event_descriptor: {
  type: "enum",
  outcomes: ["YES", "NO"]
}
```

Example for a multi-outcome market:

```
event_descriptor: {
  type: "enum",
  outcomes: ["CANDIDATE_A", "CANDIDATE_B", "CANDIDATE_C"]
}
```

> [!NOTE]
>
> Digit decomposition for numeric outcomes is out of scope for this version of the specification. Enumerated outcomes cover most common use cases.

## Oracle Announcement Format

The oracle announcement follows the [DLC specification](https://github.com/discreetlogcontracts/dlcspecs/blob/master/Messaging.md#the-oracle_announcement-type) (TLV type 55332). For uniformity with other parts of the Cashu specification, this NUT uses a JSON representation:

```json
{
  "announcement_signature": <hex_str>,
  "oracle_public_key": <hex_str>,
  "oracle_event": {
    "oracle_nonces": <Array[hex_str]>,
    "event_maturity_epoch": <int>,
    "event_descriptor": {
      "type": "enum",
      "outcomes": <Array[str]>
    },
    "event_id": <str>
  }
}
```

- `announcement_signature`: 64-byte Schnorr signature (128-character hex string)
- `oracle_public_key`: 32-byte x-only public key (64-character hex string)
- `oracle_nonces`: Array of 32-byte x-only points (each 64-character hex string)
- `event_maturity_epoch`: Unix timestamp (u32) when the event matures
- `event_descriptor`: Enumerated outcomes descriptor
- `event_id`: UTF-8 string identifier for the event

> **Note:** If an oracle does not know about this specification and publishes the event as TLV, wallets MUST deserialize and create an equivalent JSON representation.

### Announcement Signature

The `announcement_signature` is a BIP 340 Schnorr signature over the SHA256 hash of the serialized `oracle_event`, using the tag `"announcement/v0"`:

```
tag_hash = SHA256(SHA256("DLC/oracle/announcement/v0") || SHA256("DLC/oracle/announcement/v0"))
message = SHA256(tag_hash || serialize(oracle_event))
announcement_signature = BIP340_sign(oracle_sk, message)
```

### Adaptor Signature Points

For each outcome in the event descriptor, an **adaptor signature point** can be computed from the oracle's nonce. This point is used for verification:

```
adaptor_point(outcome) = R + H(outcome) * oracle_pubkey
```

Where `R` is the oracle nonce and `H(outcome)` is the tagged hash of the outcome string.

### Multi-Oracle Support

This specification supports multi-oracle schemes following the [DLC multi-oracle specification](https://github.com/discreetlogcontracts/dlcspecs/blob/master/MultiOracle.md). For a threshold `t` of `n` oracles:

- All `n` oracle announcements must be registered with the mint
- At least `t` oracles must attest for redemption
- Each oracle's adaptor point is computed independently

### Verification Requirements

When receiving proofs with the ORACLE spending condition, the mint MUST verify:

1. **Signature validity** - The `announcement_signature` is valid for the `oracle_event` using the `oracle_public_key`
2. **Event descriptor match** - The `outcome` tag value is one of the outcomes in the announcement's event_descriptor

## Witness Format

`ORACLEWitness` is a serialized JSON string of the form:

```json
{
  "oracle_sig": <hex_str>,
  "signatures": <Array[<hex_str>]>
}
```

- `oracle_sig` contains the oracle's 64-byte Schnorr signature on the attested outcome (128-character hex string)
- `signatures` contains additional P2PK signatures as per [NUT-11][11] rules (optional)

Example witness for an oracle-resolved spend:

```json
{
  "oracle_sig": "a1b2c3...64_bytes...f1e2d3",
  "signatures": []
}
```

## Spending ORACLE Proofs

A `Proof` with a `Secret` of kind `ORACLE` can be spent when the oracle attests to the outcome.

### Oracle Pathway

The holder of the correct outcome proof can spend after the oracle attests by providing:

1. An oracle signature on the outcome in `Proof.witness.oracle_sig`
2. The outcome MUST match the `outcome` tag in the secret
3. Additional signatures as per [NUT-11][11] rules if `pubkeys` are specified

## Verification Logic

The mint MUST verify the following before accepting an oracle claim:

1. Parse and validate the `announcement` tag as an oracle_announcement (JSON format)
2. Verify the `announcement_signature` using the `oracle_public_key`
3. The `outcome` tag value is one of the outcomes in the announcement's event_descriptor
4. The oracle signature `Proof.witness.oracle_sig` is valid:
   - Use the signing algorithm with tag `"attestation/v0"`
   - Message is the UTF-8 NFC-normalized outcome string from `outcome` tag
   - Verify against the `oracle_public_key` from the announcement
5. If `pubkeys` tags are present: valid signatures as per [NUT-11][11] rules
6. For multi-oracle markets: require signatures from at least `threshold` distinct oracles

### Outcome Attestation Handling

When an oracle attests to a winning outcome, the mint SHOULD check which outcome has been attested and SHOULD NOT refund the collateral for non-winning outcomes, even if a valid signature/attestation for those outcomes were somehow produced (e.g., through oracle key compromise). This prevents double-spending scenarios where an attacker with a compromised oracle key attempts to redeem multiple outcomes.

Implementation note: Mints SHOULD track which outcome has been officially attested for each event and reject redemption attempts for other outcomes after the first valid attestation is processed.

## Security Considerations

### Oracle Trust

This specification relies on oracles as the source of truth for outcome attestation:

- **Single Point of Failure**: If the oracle private key is compromised, fraudulent attestations are possible
- **Oracle Failure**: If the oracle fails to attest, users must contact the mint for resolution (see Q&A)
- **Oracle Reputation**: Users should verify the oracle's reputation before participating

**Mitigation**: Multi-oracle schemes can distribute trust across multiple oracles.

### Announcement Immutability

Once an announcement is used in ORACLE proofs:

- The oracle MUST NOT reuse nonces across different events
- If oracle nonces are reused, the private key can be extracted from multiple attestations
- Wallets SHOULD warn users if they detect nonce reuse

### Event Expiry and Vesting Period

Mint MAY remove and forget about an event if a certain period has passed since the event maturity. This "vesting period" allows users adequate time to redeem their winning outcome tokens after oracle attestation.

**Recommendation**: The vesting period SHOULD be at least 30 days after `event_maturity_epoch`. Mints MUST clearly communicate their vesting period via the [Mint Info Setting](#mint-info-setting).

After the vesting period expires:

- The mint MAY refuse to process redemptions for the expired event
- The mint MAY delete all event-related data
- Users SHOULD be aware of the vesting period and redeem before expiry

### DLC Oracle Compatibility

Using DLC-compatible signing ensures:

- Signatures from existing DLC oracles work without modification
- Oracle infrastructure can be shared between DLC and Cashu applications
- Well-tested cryptographic primitives are reused

## Q&A: Design Decisions

### Why 32-byte x-only public keys instead of 33-byte compressed?

BIP 340 and the DLC specification use x-only public keys. Using the same format ensures oracle signatures work without modification and reduces encoding overhead.

### Why not support digit decomposition for numeric outcomes?

In DLC, digit decomposition allows representing large numeric ranges with fewer oracle nonces. Having many outcomes results in many off-chain transactions (CETs).

In Cashu, the implications are different and more severe:

- Many possible outcomes = many types of outcome tokens
- This creates severe liquidity fragmentation - each outcome token needs its own market depth
- In DLCs, liquidity for each outcome is not a concern (bilateral contract)
- In ecash prediction markets, liquidity is critical for trading

**Better approach:** Create markets with fewer discrete outcomes (e.g., price ranges like "$90k-$100k" vs "$100k-$110k") and let the real-time trading price of tokens represent finer granularity.

### Why not a custom signature scheme?

Interoperability with existing DLC oracles is a key design goal. Using the exact DLC signing algorithm means any DLC oracle can be used with Cashu without modification. This expands the oracle ecosystem available to Cashu applications.

### Why is the outcome in the message, not the market ID?

Each outcome token needs to verify that the oracle attested to its specific outcome. The oracle signs each possible outcome separately, allowing only the winning outcome tokens to be spent.

### Why no locktime/refund pathway?

In DLC, the locktime/refund mechanism is essential because:

- Funds are locked in an on-chain 2-of-2 multisig
- If the oracle fails to attest, funds would be stuck forever
- Refund minimizes trust assumption against non-responsive oracles

In Cashu's custodial model, this is unnecessary:

- The mint is already trusted to honor redemptions
- If an oracle fails to attest, users can contact the mint for resolution
- The mint can administratively resolve disputed or abandoned markets
- No funds are "stuck on-chain" - the mint can always intervene

Removing these tags simplifies implementation without increasing risk in Cashu's trust model.

## API Endpoints

### Register Oracle Announcement

Mints MAY provide an endpoint for registering oracle announcements. This is useful when oracles and mints are separate entities.

```
POST /v1/oracle/announcements
```

Request:

```json
{
  "announcements": <Array[hex_str]>
}
```

Response:

```json
{
  "registered": <int>,
  "verified": <Array[{
    "oracle_pubkey": <hex_str>,
    "verified": <bool>,
    "adaptor_points": {
      "<outcome_1>": <hex_str>,
      "<outcome_2>": <hex_str>
    }
  }]>
}
```

- `registered`: Number of announcements successfully registered
- `verified`: Array of verification results with computed adaptor points for each outcome

### Get Announcements

```
GET /v1/oracle/announcements/{event_id}
```

Response:

```json
{
  "announcements": <Array[{
    "oracle_pubkey": <hex_str>,
    "announcement": <hex_str>,
    "verified": <bool>,
    "adaptor_points": {
      "<outcome_1>": <hex_str>,
      "<outcome_2>": <hex_str>
    }
  }]>
}
```

## Mint Info Setting

The [NUT-06][06] `MintMethodSetting` indicates support for this feature:

```json
{
  "28": {
    "supported": true,
    "vesting_period": <int>
  }
}
```

- `supported`: Boolean indicating NUT-28 support
- `vesting_period`: (optional) Number of seconds after `event_maturity_epoch` during which the mint will honor redemptions. If not specified, implementations SHOULD assume a minimum of 30 days (2592000 seconds). A value of `0` indicates no expiry.
