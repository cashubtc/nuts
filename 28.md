# NUT-28: Oracle Spending Condition

`optional`

`depends on: NUT-10`

---

This NUT describes an oracle spending condition that allows ecash proofs to be locked to an oracle-attested outcome. Using this spending condition, proofs can only be spent after an oracle publishes a signature attesting to a specific outcome. This enables use cases such as prediction markets, event-based tokens, and oracle-resolved conditional payments.

The oracle signature scheme is designed to be compatible with the [DLC (Discreet Log Contracts) specification](https://github.com/discreetlogcontracts/dlcspecs), allowing Cashu mints to leverage existing DLC oracle infrastructure.

Caution: Applications that rely on oracle resolution must verify that the oracle is trustworthy and check via the mint's [info][06] endpoint that NUT-28 is supported.

Caution: If the mint does not support this type of spending condition, proofs may be treated as regular anyone-can-spend proofs. Applications must ensure that the mint supports a specific kind of spending condition by checking the mint's [info][06] endpoint.

## Motivation

Oracle spending conditions enable powerful use cases where ecash is released based on real-world events. By designing for DLC compatibility, this specification allows:

1. **Reuse of existing oracle infrastructure** - Existing DLC oracles can be used without modification
2. **Standardized signature verification** - Well-tested BIP 340 Schnorr signatures with tagged hashing
3. **Accountability** - Optional oracle announcements provide cryptographic proof of oracle commitments

## ORACLE Locked Proof

[NUT-10][10] Secret `kind: ORACLE`

If for a `Proof`, `Proof.secret` is a `Secret` of kind `ORACLE`, the oracle public key and event information are stored in the secret's tags, and the oracle signature for unlocking the proof is provided in `Proof.witness.oracle_sig`.

Here is a concrete example of a `Secret` of kind `ORACLE`:

```json
[
  "ORACLE",
  {
    "nonce": "da62796403af76c80cd6ce9153ed3746",
    "data": "",
    "tags": [
      [
        "oracle",
        "9be6fa256a022aafc98f24a71f0e37ab2ac6fe5b208a77a3d429b4b5c59f7ce0"
      ],
      ["event", "btc_price_100k_2025"],
      ["outcomes", "YES", "NO"],
      ["outcome", "YES"],
      ["threshold", "1"]
    ]
  }
]
```

The oracle public key in the `oracle` tag uses the **32-byte x-only format** (as defined in [BIP 340](https://github.com/bitcoin/bips/blob/master/bip-0340.mediawiki)), encoded as a 64-character hexadecimal string.

## Tags

The following tags define the oracle spending condition:

### Required Tags

- `oracle: <hex_str>` - The 32-byte x-only public key of the oracle (64 hex characters)
- `event: <str>` - A unique identifier for the event being attested
- `outcomes: <str>...` - The possible outcomes for this event (one or more values). This defines the event descriptor.
- `outcome: <str>` - The specific outcome this proof is locked to. MUST be one of the values in `outcomes`.

### Optional Tags

- `threshold: <int>` - Minimum number of oracle signatures required (default: 1). Enables multi-oracle schemes when combined with multiple `oracle` tags.
- `maturity: <int>` - Unix timestamp after which the oracle may attest. The mint MUST reject spends before this time.
- `locktime: <int>` - Unix timestamp after which the refund pathway becomes available
- `refund: <hex_str>` - Public keys that can spend after `locktime` (_allows multiple entries_). Uses 33-byte compressed format.
- `announcement: <hex_str>` - Optional DLC oracle announcement for accountability (see [Oracle Announcement](#oracle-announcement-optional))

Optional P2PK tags from [NUT-11][11] may also be used:

- `sigflag: <str_enum[SIG_FLAG]>` - Sets the signature flag
- `pubkeys: <hex_str>` - Additional public keys for multisig verification (_allows multiple entries_)
- `n_sigs: <int>` - Minimum number of public keys providing valid signatures

## Oracle Signature Scheme

The oracle signature scheme follows the [DLC specification](https://github.com/discreetlogcontracts/dlcspecs/blob/master/Oracle.md#signing-algorithm) for maximum compatibility with existing DLC oracle infrastructure.

### Tagged Hash

Signatures use BIP 340 tagged hashing. The `tag_hash` function is defined as:

```
tag_hash(tag) = SHA256(SHA256(tag) || SHA256(tag))
```

Where `||` denotes byte concatenation.

### Signing Algorithm

The algorithm `Sign(sk, message, tag)` is defined as:

```
Sign(sk, message, tag):
  H = tag_hash("DLC/oracle/" || tag)
  m = H(message)
  return BIP340_sign(sk, m)
```

For attestations, the tag is `"attestation/v0"`, so the full tag string is `"DLC/oracle/attestation/v0"`.

### Message Format

The message to be signed is the **UTF-8 encoded outcome string with NFC normalization**. For example, if the outcome is `"YES"`, the message is the UTF-8 bytes of `"YES"`.

### Complete Signing Example

To sign the outcome `"YES"`:

1. Normalize the outcome string to NFC: `"YES"` (already normalized)
2. Compute the tagged hash:
   - `tag = "DLC/oracle/attestation/v0"`
   - `tag_hash = SHA256(SHA256(tag) || SHA256(tag))`
3. Compute the message hash:
   - `m = SHA256(tag_hash || UTF8("YES"))`
4. Sign with BIP 340:
   - `sig = BIP340_sign(sk, m)`

### Verification

To verify an oracle signature:

```
Verify(pk, message, sig, tag):
  H = tag_hash("DLC/oracle/" || tag)
  m = H(message)
  return BIP340_verify(pk, m, sig)
```

Where `pk` is the 32-byte x-only public key from the `oracle` tag.

## Event Descriptor

The `outcomes` tag defines the enumerated outcomes for an event, equivalent to the DLC `enum_event_descriptor`. This is a simple enumeration where each outcome is a UTF-8 string with NFC normalization.

Example for a binary market:

```json
["outcomes", "YES", "NO"]
```

Example for a multi-outcome market:

```json
["outcomes", "CANDIDATE_A", "CANDIDATE_B", "CANDIDATE_C"]
```

> [!NOTE]
>
> Digit decomposition for numeric outcomes is out of scope for this version of the specification. Enumerated outcomes cover most common use cases.

## Oracle Announcement (Optional)

For accountability and DLC compatibility, an oracle announcement can be included in the `announcement` tag. When present, wallets and mints SHOULD verify the announcement signature.

The announcement follows the DLC [oracle_announcement](https://github.com/discreetlogcontracts/dlcspecs/blob/master/Messaging.md#the-oracle_announcement-type) format (TLV type 55332):

- `announcement_signature`: 64-byte Schnorr signature on the oracle_event
- `oracle_public_key`: 32-byte x-only public key
- `oracle_event`: Contains nonces, maturity epoch, event descriptor, and event ID

The announcement signature uses the tag `"announcement/v0"`.

When an announcement is present:

1. Verify the announcement signature matches the oracle public key
2. Verify the event descriptor matches the `outcomes` tag
3. Verify the event ID matches the `event` tag

## Witness Format

`ORACLEWitness` is a serialized JSON string of the form:

```json
{
  "oracle_sig": <hex_str>,
  "signatures": <Array[<hex_str>]>
}
```

- `oracle_sig` contains the oracle's 64-byte Schnorr signature on the attested outcome (128-character hex string)
- `signatures` contains additional P2PK signatures as per [NUT-11][11] rules (optional)

Example witness for an oracle-resolved spend:

```json
{
  "oracle_sig": "a1b2c3...64_bytes...f1e2d3",
  "signatures": []
}
```

## Spending ORACLE Proofs

A `Proof` with a `Secret` of kind `ORACLE` can be spent in two ways.

### Oracle Pathway (oracle signature)

The holder of the correct outcome proof can spend after the oracle attests by providing:

1. An oracle signature on the outcome in `Proof.witness.oracle_sig`
2. The outcome MUST match the `outcome` tag in the secret
3. If `maturity` tag is present, current time MUST be >= `maturity`
4. Additional signatures as per [NUT-11][11] rules if `pubkeys` are specified

### Refund Pathway (timelocked refund)

If the `refund` tag is present and the `locktime` has passed:

1. The signer(s) listed in the `refund` tag can spend the proof
2. Requires valid signature(s) as per [NUT-11][11] Refund MultiSig rules

This pathway handles cases where the oracle fails to attest, the event is cancelled, or there are disputes.

> [!NOTE]
>
> If the `refund` tag is not present, the ORACLE proof can only be spent via the oracle pathway. If `locktime` passes without a `refund` tag, the proof remains locked.

## Verification Logic

### Oracle Claims

The mint MUST verify the following before accepting an oracle claim:

1. If `maturity` tag is present: current time >= `maturity`
2. The `outcome` tag value is one of the values in the `outcomes` tag
3. The oracle signature `Proof.witness.oracle_sig` is valid:
   - Use the signing algorithm with tag `"attestation/v0"`
   - Message is the UTF-8 NFC-normalized outcome string from `outcome` tag
   - Verify against the public key in the `oracle` tag
4. If `announcement` tag is present: verify announcement signature (SHOULD)
5. If `pubkeys` tags are present: valid signatures as per [NUT-11][11] rules
6. If `threshold` > 1: require signatures from `threshold` distinct oracles

### Refund Claims

The mint MUST verify the following before accepting a refund claim:

1. The `refund` tag is present
2. Current time > `locktime`
3. Valid refund signature(s) as per [NUT-11][11] Refund MultiSig rules

## Security Considerations

### Oracle Trust

This specification relies on oracles as the source of truth for outcome attestation:

- **Single Point of Failure**: If the oracle private key is compromised, fraudulent attestations are possible
- **Oracle Failure**: If the oracle fails to attest, tokens can only be redeemed via the refund pathway
- **Oracle Reputation**: Users should verify the oracle's reputation before participating

**Mitigation**: Multi-oracle schemes using the `threshold` tag can distribute trust across multiple oracles.

### Time Synchronization

The mint relies on its local clock for `maturity` and `locktime` constraints. Clock skew may cause validation issues.

**Mitigation**: Mints should use NTP-synchronized time. Wallets should allow a small grace period near time boundaries.

### DLC Oracle Compatibility

Using DLC-compatible signing ensures:

- Signatures from existing DLC oracles work without modification
- Oracle infrastructure can be shared between DLC and Cashu applications
- Well-tested cryptographic primitives are reused

## Q&A: Design Decisions

### Why not mandatory oracle announcements?

Oracle announcements provide accountability but add complexity. For simpler use cases (trusted oracle, known event), the announcement is unnecessary overhead. Unlike DLCs which require adaptor signatures, Cashu's custodial model doesn't strictly need the cryptographic commitments that announcements provide.

### Why 32-byte x-only public keys instead of 33-byte compressed?

BIP 340 and the DLC specification use x-only public keys. Using the same format ensures oracle signatures work without modification and reduces encoding overhead.

### Why not support digit decomposition for numeric outcomes?

Digit decomposition adds significant complexity for both specification and implementation. Enumerated outcomes cover the vast majority of prediction market use cases. Numeric decomposition can be added in a future extension if needed.

### Why not a custom signature scheme?

Interoperability with existing DLC oracles is a key design goal. Using the exact DLC signing algorithm means any DLC oracle can be used with Cashu without modification. This expands the oracle ecosystem available to Cashu applications.

### Why is the outcome in the message, not the market ID?

Each outcome token needs to verify that the oracle attested to its specific outcome. The oracle signs each possible outcome separately, allowing only the winning outcome tokens to be spent.
