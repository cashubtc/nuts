# NUT-29: Conditional Token Framework

`optional`

`depends on: NUT-28`

---

This NUT describes a Conditional Token Framework (CTF) that enables prediction markets using ecash. Inspired by the [Gnosis Conditional Token Framework](https://docs.gnosis.io/conditionaltokens/), this specification allows users to deposit collateral and receive complete sets of outcome tokens for a market. Outcome tokens are locked using the [NUT-28][28] oracle spending condition and can be redeemed after the oracle attests to the winning outcome.

Caution: This specification relies on [NUT-28][28] for oracle attestation. Applications must verify that the mint supports both NUT-28 and NUT-29 by checking the mint's [info][06] endpoint.

## Motivation

Conditional tokens enable powerful prediction market use cases:

1. **Complete outcome sets** - Users receive tokens for ALL possible outcomes, ensuring the market is fully collateralized
2. **Trading** - Outcome tokens can be freely traded before resolution
3. **Atomic splits** - A single operation creates all outcome tokens, preventing partial positions
4. **Unified redemption** - Winners redeem via standard swap operations using NUT-28

By following the Gnosis CTF model, this specification provides a proven approach to conditional tokens adapted for the Cashu ecosystem.

## Alternative: Single Token Issuance (NUT-28 Only)

Prediction markets can be implemented using only [NUT-28][28] without the CTF split operation. In this approach:

1. **Single token issuance** - The mint issues tokens for individual outcomes directly, rather than complete sets
2. **AMM-based pricing** - The mint integrates an Automated Market Maker (AMM) to determine current prices (odds) for each outcome token
3. **Simpler liquidity** - Liquidity is managed by the mint's AMM rather than requiring users to hold complete sets

**Trade-offs:**

- **Pros**: Simpler user experience, no need to manage complete sets
- **Cons**: Requires AMM integration in the mint, potential liquidity fragmentation for conditional/combinatorial markets where the same collateral backs multiple related markets

The CTF approach (this NUT) is preferred when:

- Users want to create their own positions by splitting collateral
- Combinatorial markets require consistent liquidity across conditions
- Decentralized trading without mint-operated AMM is desired

## Overview

The CTF lifecycle consists of:

1. **Split**: User deposits collateral → receives complete set of outcome tokens
2. **Trade**: Users trade outcome tokens (normal Cashu transfers)
3. **Oracle Attests**: Oracle signs the winning outcome
4. **Redeem**: Winners swap their tokens using oracle signature (via NUT-28)

```
           Split                    Trade                 Attest              Redeem
User ──────────────► Outcome    ◄────────────►    Oracle ────────►    Winner ──────────►
     100 sats        Tokens         P2P           Signs              Swaps for
                     (YES+NO)       Trading       Outcome            Collateral
```

## Market ID

A market is uniquely identified by a `market_id` computed from the market parameters. This follows the Gnosis CTF condition ID pattern, using SHA256 instead of keccak256 for Bitcoin ecosystem consistency.

### Market ID Calculation

```
market_id = SHA256(sorted_oracle_pubkeys || sorted_event_id || outcome_count)
```

Where:

- `sorted_oracle_pubkeys`: One or more 32-byte x-only public keys of the oracles (same format as NUT-28), sorted lexicographically (byte-wise) and concatenated. For multi-oracle markets with threshold `t` of `n` oracles, all `n` oracle pubkeys are included. Derived from `announcements[].oracle_public_key`.
- `sorted_event_id`: UTF-8 encoded event identifier string (called "question_id" in the original Gnosis CTF context). For combinatorial markets involving multiple conditions, condition identifiers MUST be sorted lexicographically (byte-wise) and concatenated with a null byte (0x00) separator before encoding. Derived from `announcements[].oracle_event.event_id`.
- `outcome_count`: 1-byte unsigned integer representing the number of outcomes. Derived from `len(announcements[0].oracle_event.event_descriptor.outcomes)`.

The `market_id` is a 32-byte value encoded as a 64-character hexadecimal string. Since all components are derived from the announcements, the `market_id` can be computed by either the wallet or the mint from the announcement data.

**Rationale for sorting:**

- **Oracle sorting:** In multi-oracle schemes, the market_id must be computed canonically regardless of the order oracles are specified. Sorting ensures `[oracleA, oracleB]` and `[oracleB, oracleA]` produce the same market_id.
- **Condition sorting:** In combinatorial markets where positions depend on multiple conditions, sorting ensures `(conditionA, conditionB)` and `(conditionB, conditionA)` produce the same market_id.

### Example Calculation

**Single oracle, single condition:**

```
oracle_pubkey:  9be6fa256a022aafc98f24a71f0e37ab2ac6fe5b208a77a3d429b4b5c59f7ce0
event_id:       "btc_price_100k_2025"
outcome_count:  2

market_id = SHA256(
  0x9be6fa256a022aafc98f24a71f0e37ab2ac6fe5b208a77a3d429b4b5c59f7ce0 ||
  "btc_price_100k_2025" ||
  0x02
)
```

**Multiple oracles (threshold 2-of-3):**

```
oracle_pubkeys = ["9be6fa...", "79be66...", "a1b2c3..."]
sorted_oracle_pubkeys = ["79be66...", "9be6fa...", "a1b2c3..."]  // sorted lexicographically

market_id = SHA256(
  0x79be66... || 0x9be6fa... || 0xa1b2c3... ||
  "btc_price_100k_2025" ||
  0x02
)
```

**Multiple conditions (combinatorial market):**

```
conditions = ["election_winner", "btc_price"]
sorted_event_id = "btc_price" + 0x00 + "election_winner"  // sorted lexicographically

market_id = SHA256(
  0x9be6fa... ||
  0x6274635f7072696365006f6c656374696f6e5f77696e6e6572 ||  // "btc_price\x00election_winner"
  0x04  // 4 outcomes for 2x2 combinatorial
)
```

## Split Operation

The split operation allows users to deposit collateral and receive a complete set of outcome tokens. For every unit of collateral deposited, the user receives one token for each possible outcome.

Markets must be registered via `POST /v1/ctf/markets` before splitting.

### Endpoint

```
POST /v1/ctf/split
```

### Request Body

```json
{
  "market_id": <hex_str>,
  "inputs": <Array[Proof]>,
  "outputs": {
    "<outcome_1>": <Array[BlindedMessage]>,
    "<outcome_2>": <Array[BlindedMessage]>,
    ...
  }
}
```

- `market_id`: The 32-byte market identifier (64 hex characters). Must reference a market registered via `POST /v1/ctf/markets`. Returns error 13021 (Market not found) if unknown.
- `inputs`: Array of `Proof` objects used as collateral (see [NUT-00][00])
- `outputs`: Object mapping each outcome to an array of `BlindedMessage` objects

The `unit` for the market is derived from the input proofs' keyset.

If the mint returns error 13021 (Market not found), the wallet SHOULD register the market using `POST /v1/ctf/markets` and retry the split.

### Output Requirements

1. The total amount of `inputs` MUST equal the total amount of EACH outcome's outputs
2. All outcomes defined in the announcements' event descriptor MUST be present in `outputs`
3. Each outcome's output amount MUST be identical

For example, with 100 sats collateral in a binary market:

- `inputs`: 100 sats total
- `outputs["YES"]`: 100 sats total in blinded messages
- `outputs["NO"]`: 100 sats total in blinded messages

### Mint Behavior

When processing a split request, the mint:

1. Looks up the market by `market_id`
2. Returns error 13021 (Market not found) if the market is not registered
3. Validates input/output amounts and completeness
4. Signs the blinded messages for each outcome

### Response Body

```json
{
  "signatures": {
    "<outcome_1>": <Array[BlindSignature]>,
    "<outcome_2>": <Array[BlindSignature]>,
    ...
  }
}
```

- `signatures`: Object mapping each outcome to an array of `BlindSignature` objects

Each `BlindSignature` corresponds to a `BlindedMessage` in the request, creating NUT-28 ORACLE-locked proofs.

### Token Structure

The mint creates outcome tokens as NUT-28 ORACLE-locked proofs. For each output, the resulting proof will have a secret of kind `ORACLE` with:

- `announcement` tag: The JSON-encoded oracle_announcement for this market (see [NUT-28][28] Oracle Announcement Format). Contains oracle pubkey, event ID, outcomes, maturity, and nonces.
- `outcome` tag: The specific outcome this token represents

Oracle pubkey, event ID, outcomes, and maturity are all derived from the announcement.

The wallet MUST construct valid NUT-10 secrets for each blinded message, embedding the correct oracle spending condition.

## Merge Operation

The merge operation allows users to combine a complete set of outcome tokens back into collateral. This is the inverse of split: for every unit of each outcome token surrendered, the user receives one unit of collateral.

The market must have been created via a prior split operation. Use the `market_id` returned from the split response.

### Endpoint

```
POST /v1/ctf/merge
```

### Request Body

```json
{
  "market_id": <hex_str>,
  "inputs": {
    "<outcome_1>": <Array[Proof]>,
    "<outcome_2>": <Array[Proof]>,
    ...
  },
  "outputs": <Array[BlindedMessage]>
}
```

- `market_id`: The 32-byte market identifier (64 hex characters). Must reference a market created via prior split operation. Returns error 13021 (Market not found) if unknown.
- `inputs`: Object mapping each outcome to an array of `Proof` objects (NUT-28 ORACLE proofs)
- `outputs`: Array of `BlindedMessage` objects for the collateral to receive

### Input Requirements

1. All outcomes defined for the market MUST be present in `inputs`
2. Each outcome's input amount MUST be identical
3. The `outputs` total MUST equal the amount of any single outcome's inputs

For example, in a binary market with 100 sats of each outcome token:

- `inputs["YES"]`: 100 sats total in ORACLE proofs
- `inputs["NO"]`: 100 sats total in ORACLE proofs
- `outputs`: 100 sats total in blinded messages (collateral returned)

### Response Body

```json
{
  "signatures": <Array[BlindSignature]>
}
```

Each `BlindSignature` corresponds to a `BlindedMessage` in the request, creating regular (non-outcome-locked) proofs.

### Merge Verification

The mint MUST verify:

1. All outcome tokens are valid NUT-28 ORACLE proofs for the specified market
2. All outcomes for the market are present
3. All outcome amounts are equal
4. Output amount equals the per-outcome input amount

## Market Registry

Markets are registered via `POST /v1/ctf/markets` before any split operations. The mint maintains a registry of markets with their parameters.

### Market Info

```json
{
  "market_id": <hex_str>,
  "threshold": <int>,
  "description": <str>,
  "announcements": <Array[OracleAnnouncement]>
}
```

- `market_id`: Computed market identifier (derived from announcements, see [Market ID Calculation](#market-id-calculation))
- `threshold`: Minimum number of oracles required for attestation (default: 1)
- `description`: Human-readable market description
- `announcements`: Array of oracle announcements in JSON format (see [NUT-28][28] Oracle Announcement Format)

The following fields can be derived from the announcements and are not included in the Market Info:

- `oracle_pubkeys`: From `announcements[].oracle_public_key`
- `event_id`: From `announcements[].oracle_event.event_id`
- `outcomes`: From `announcements[].oracle_event.event_descriptor.outcomes`
- `maturity`: From `announcements[].oracle_event.event_maturity_epoch`

The `unit` for tokens in a market is determined by the keyset used during the first split operation.

### Get Markets

```
GET /v1/ctf/markets
```

Returns an array of available markets:

```json
{
  "markets": <Array[MarketInfo]>
}
```

### Get Market

```
GET /v1/ctf/markets/{market_id}
```

Returns details for a specific market:

```json
{
  "market": <MarketInfo>
}
```

### Register Market

```
POST /v1/ctf/markets
```

Registers a new market without performing a split. This allows wallets to pre-register markets before splitting, reducing payload size for subsequent split operations. This is analogous to the `prepareCondition` operation in the Gnosis CTF.

#### Request Body

```json
{
  "threshold": <int>,
  "description": <str>,
  "announcements": <Array[OracleAnnouncement]>
}
```

- `threshold`: Minimum number of oracles required for attestation (default: 1)
- `description`: Human-readable market description
- `announcements`: Array of oracle announcements in JSON format (see [NUT-28][28] Oracle Announcement Format)

#### Response Body

```json
{
  "market_id": <hex_str>
}
```

- `market_id`: The 32-byte market identifier computed from the announcements (64 hex characters)

#### Mint Behavior

1. Computes `market_id` from the provided announcements
2. If market already exists with matching configuration: returns existing `market_id` (idempotent)
3. If market already exists with different configuration: returns error 13028
4. If market is new: verifies announcement signatures, stores configuration, returns `market_id`

#### Errors

- 13011: Oracle announcement verification failed
- 13028: Market already exists (with different configuration)

## Redemption

There is no separate redemption endpoint. Winners redeem their outcome tokens using the standard [swap][03] operation with a NUT-28 oracle signature witness.

### Redemption Flow

1. Oracle attests to winning outcome (publishes signature)
2. Winner obtains oracle signature
3. Winner submits swap request with ORACLE proofs and oracle signature in witness
4. Mint verifies oracle signature per NUT-28 rules
5. Mint returns new proofs (regular ecash, not outcome-locked)

This leverages existing swap infrastructure and NUT-28 verification logic.

## Error Codes

| Code  | Description                     |
| ----- | ------------------------------- |
| 13020 | Invalid market ID               |
| 13021 | Market not found                |
| 13022 | Split amount mismatch           |
| 13023 | Missing outcome in outputs      |
| 13024 | Market not active               |
| 13025 | Merge amount mismatch           |
| 13026 | Missing outcome in merge inputs |
| 13027 | Oracle threshold not met        |
| 13028 | Market already exists           |

Alice now holds:

- 100 sats worth of YES tokens (ORACLE-locked to outcome "YES")
- 100 sats worth of NO tokens (ORACLE-locked to outcome "NO")

### Step 3: Trading

Alice believes YES will win, so she sells her NO tokens to Bob for 40 sats. This is a normal Cashu token transfer using NUT-28 ORACLE proofs.

### Step 4: Oracle Attestation

On July 1, 2025, the oracle attests that YES won by publishing a signature:

```
outcome = "YES"
signature = Sign(sk, "YES", "attestation/v0")
```

### Step 5: Winner Redemption

Alice redeems her YES tokens via standard swap:

```json
{
  "inputs": [
    {
      "amount": 64,
      "id": "009a1f293253e41e",
      "secret": "[\"ORACLE\",{\"nonce\":\"...\",\"data\":\"\",\"tags\":[[\"announcement\",\"{\\\"announcement_signature\\\":\\\"9a0b85c6...\\\",\\\"oracle_public_key\\\":\\\"9be6fa...\\\",\\\"oracle_event\\\":{...}}\"], [\"outcome\",\"YES\"]]}]",
      "C": "02...",
      "witness": "{\"oracle_sig\":\"...\",\"signatures\":[]}"
    }
    // ... more proofs
  ],
  "outputs": [
    // Regular blinded messages for new proofs
  ]
}
```

The `announcement` tag contains the JSON-encoded oracle announcement (escaped as a JSON string within the secret).

The mint verifies the oracle signature per NUT-28 and returns regular (non-outcome-locked) proofs.

## Security Considerations

### Collateral Safety

The mint holds all collateral. The total value of all outcome tokens for a market equals the collateral deposited. When the oracle attests:

- Winning outcome tokens can be redeemed for collateral
- Losing outcome tokens become worthless (cannot be spent)

### Market Integrity

1. **Complete Sets**: Split always creates ALL outcomes, preventing partial collateralization
2. **Equal Amounts**: Each outcome receives equal token amounts
3. **Oracle Independence**: The mint does not determine outcomes; the oracle does

### Trust Model

This specification inherits Cashu's trust model:

- Users trust the mint to honor redemptions
- Users trust the oracle to attest honestly
- The mint does not need to trust users (cryptographic verification)
- **The mint does not need to trust the oracle** - Whether using AMM-style or CTF-style prediction markets, the mint has no counterparty risk from oracle outcomes. The mint's collateral is always matched by outstanding outcome tokens. Risk of deceptive oracles lies entirely with the users who must verify oracle reputation before participating.

## Q&A: Design Decisions

### Why not keccak256 for market ID?

SHA256 is the standard hash function in the Bitcoin and Cashu ecosystem. Using SHA256 maintains consistency and avoids adding keccak256 as a dependency.

### When should users merge vs. wait for resolution?

Merge is useful when:

- A user holds a complete set and wants to exit their position before oracle attestation
- Market conditions change and the user wants to recover collateral immediately
- Arbitrage opportunities exist between the market price and collateral value

Waiting for resolution is simpler when:

- The user expects one outcome to win and wants to maximize profit
- Transaction fees make merge uneconomical

### Why no separate redeem endpoint?

NUT-28's oracle spending condition handles redemption via the standard swap endpoint. This reuses existing infrastructure and keeps the specification minimal. Winners simply swap their ORACLE-locked proofs with the oracle signature as witness.

### Why require the wallet to construct secrets?

The wallet constructs NUT-10 secrets with the correct ORACLE spending conditions. This maintains the standard Cashu flow where wallets create secrets and the mint blindly signs. The mint verifies the secret structure when the proof is spent.
